My approach to problem-solving

Tell your non-tech friend a story about a time you:

1. were blocked on a simple problem:

I was stuck on a simple problem of forgetting to add one character to make the code work properly. To solve this problem, I used the rubber duck method. To use this method, I read out each line of code as if I were explaining it to a rubber duck. Eventually, it forced me to read out the error I had made.


Throughout the process, I felt frustrated and annoyed. I knew there was a problem. I knew it was subtle problem. But I couldn’t find it for a while. After this experience, I learned that frustration doesn’t help at all. As soon as you face a problem, turn to a problem-solving method to not waste energy on feeling frustrated.

2. elegantly solved a problem.

The problem was to create a function that accepts an array of numbers, but only returns the even numbers in a new array. I used the problem-solving technique of pseudocode. This means that I wrote out the logic to make the function work in an informal way. That way, my mind doesn’t get bogged down with the nitty gritty details of JavaScript syntax.


Throughout the process, I felt really good. For some reason, the problem just unravelled very smoothly as I wrote it out in pseudocode. And then when I used the pseudocode to write out the JavaScript, everything worked the first time.


I learned that sometimes a solution to a problem will come easy, and sometimes it won’t. Because of this, it’s easy to feel frustrated when you’re stuck on a problem, but right before you solved a different problem with ease. So, whenever you fail a problem, you should view it as “not solved yet”, rather than “failed”. That way, you’ll minimise frustration when you get blocked.

3. Reflect on how confident you feel using each of these problem-solving techniques/processes:


Pseudocode

I felt very confident about using Pseudocode, because I’ve had experience with it before.



Trying something

I’m not completely sure what the meaning of this method is. But sometimes I do trying something new when my current solution doesn’t work. It’s always a bit of a gamble of trial and error. So I don’t rely on it often.



Rubber ucky method

Using the rubber ducky method is a last resort for me. This is because it is quite a tedious process to read every line of code. However, when I have used it, it has worked very well for me.



Reading error messages

Reading error messages can often times be frustrating for me, as it feels I’m just staring at my own failure. But once I rearrange my mind to see an error message as a pathway to a solution, then they can be very helpful for me.



Console.logging

Console.logging works excellently for me. It’s my first go-to problem solving technique when something doesn’t work as expected to. It has revealed countless issues in my code for me.



Googling

Googling works for solving any problem in life, whether it be how to cook something, or finding a product you want to buy, or how to code. It’s the best source of knowledge in the world. So it’s a no brainer to use Google to help you with coding. It’s amazing and I can’t imagine a life without it.



Asking your peers for help

Asking a peer for help is a good way of getting a fresh perspective on your code. Sometimes if you look at a problem for too long, you won’t be able to look at the problem in an insightful way. By asking a peer, they can provide an insight to your code and help you solve the problem.



Asking coaches for help

I never had to ask a coach for help. I’ve luckily not come across a problem yet that I couldn’t solve on my own.



Improving your process with reflection

I’ve used this process whenever I’ve overcome a roadblock in my code. It’s a great way to learn from your mistakes so you don’t repeat them in the future, although sometimes I don’t remember to be intentional about it. I just hope that my subconscious remembers the mistakes I’ve made, and will warn me next time I’m about to make the same mistake again.


4. Reflect on a time you were reluctant to ask for help. Consider what made you reluctant to do so.

At one point I thought that the test wasn’t working properly in the JavaScript Kata challenges. I was reluctant to reach out to a facilitator because I felt a risk of feeling silly for blaming the issue on the testing, and not my own code.
